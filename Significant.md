# 问题

不知道项目本质是什么，要做什么，怎么做，只是跟着计划盲目一步一步做，
但这个危险的根源不是"跟着AI写代码"

我来给你建立这个心智模型，只用三行字：

用户消息 → [你的程序问LLM"该怎么回复"] → 回复发出去

这就是整个项目的本质。你现在写的每一个文件，都是在为这三步的某一个环节服务：

config.py   → 提供连接 LLM 和 Telegram 需要的密码
llm.py      → 实现"问LLM"这一步
bot.py      → 实现"收用户消息"和"发回复"这两步
agent.py    → 决定"什么时候直接问LLM、什么时候先搜索、什么时候先查记忆"
session.py  → 让LLM知道"之前聊了什么"
tools/      → 搜索网络、查记忆的具体实现
memory/     → 跨对话存储和检索记忆

如果你在写每一行代码之前，能把这个文件在那三步里的位置说出来，你就有了足够的全局模型去理解你在做什么。

你的学习流程里缺少一个关键步骤

现在你的流程是：
计划说"这周写llm.py" → 和AI对话 → AI给代码 → 文件存在了 → 继续下一步

缺少的那一步在这里：
计划说"这周写llm.py" → 和AI对话 → AI给代码 → [文件存在了]
                                                        ↓
                                           关掉AI，新建一个空白文件
                                           不看已有代码，自己试着写
                                           写不出来 → 找到写不出来的那一行
                                           只针对那一行问AI"这行为什么这样写"
                                           直到能写出来 → 才算学会

这一步叫主动提取（active recall），是唯一能区分"持有代码"和"理解代码"的动作。

你不需要能独立设计整个 llm.py，但你需要能做到：合上所有参考，写出 async def ask(prompt) 的基本骨架，并能解释每一行的目的。

如果现在做不到这件事，你需要在继续月2之前把它做到。不是因为计划要求，而是因为 llm.py 是整个项目里所有文件的基础——它的逻辑你如果不懂，后面没有一个文件你会真正懂。

第一攻：列出"需要改进的地方"容易变成一堆抽象建议——"一次只问一个问题"这种话谁都会说，但不会改变行为。真正有价值的总结是：找到我们对话里反复出现的具体失败模式，说清楚每次在哪个节点出了问题。

第二攻：攻击第一攻——找具体失败模式还不够深。所有问题都有一个共同根源：用户持有一个错误信念——"听懂了解释 = 能写出代码"。这是假的。听懂和能写是两件完全不同的事，中间差了一个"自己动手试"的步骤。如果这个信念不改，换什么问法、换什么解释方式都治标不治本。

第三攻：攻击"双方各自改进"的对称框架——用户的问题和我的问题不对等。我的问题（解释太多、行动太少）是可以立刻改的；用户的问题（不在解释之间动手）是习惯问题，需要结构来约束，而不只是意识到。最有用的结论是：我来提供约束结构，用户在结构里执行。

一、你的学习方式里有问题的地方

问题1：听懂了解释，但不在解释之间动手

[整个llm.py](http://整个llm.py)的学习过程里，我解释了十几个概念，但你实际动手写代码的次数极少。每次我说"去改这个，改完给我看"，你通常带着新问题回来，而不是带着修改后的代码回来。

这不是态度问题，是对"学会"的定义有误。听懂 ≠ 学会。学会的唯一检验是：合上参考，自己能写出来。

具体表现：

- [test.py](http://test.py)被要求修改题2，多次

- [llm.py](http://llm.py)的框架给出来之后，你"我不知道怎么填"

- 每次我给解释，你读完就问下一个问题，没有先试着用

问题2：一次发4-5个问题

你的消息通常包含3-5个独立问题，覆盖不同概念。这导致：

- 我需要回答很多内容，回复变得很长

- 你读完长回复之后，一个都没有真正吸收就继续问下一批

- 问题之间没有递进关系，是平行发出的

问题3："现阶段是否需要弄明白"问得太频繁

这个问题在我们对话里出现了至少5次。它反映的焦虑是真实的，但也在消耗时间。大部分情况下，答案是：先记模式，做完项目之后回来理解。 这个默认答案可以直接用，不需要每次都问我。

问题4：没有自己先尝试再问

你的提问模式通常是"我不知道X怎么写"，而不是"我试着写了X，是这样的……，但不对"。这两种提问的效果完全不同。前者让我无法知道你缺的是什么，后者让我能精准指出问题在哪里。

二、你应该怎么问问题、怎么对话

规则1：一次只问一个问题

把4个问题拆成4次对话，每次问一个，等我回答，然后立刻去试，再来问下一个。

规则2：先试再问

在问"X怎么写"之前，先写一个你认为可能对的版本，哪怕你觉得它是错的。

- 不好的问法：response那一行怎么写

- 好的问法：我觉得response那行应该写response = [client.chat](http://client.chat).create(prompt)，

这样我能直接指出你理解对了什么、错了什么，比从头解释快得多。

规则3：解释之后立刻动手，动

每次我给了解释或者叫你去做某件事，关掉对话，去做，做完再开对话。不要在读完解释之后立刻问下一个问题。

规则4："现阶段是否需要弄明白"的默认答案

[除非你问的是llm.py](http://除非你问的是llm.py)里的核心逻辑（比如为什么用async，为什么有messages），其他细节默认答案都是记模式，继续走，以后回来理解。不需要每次问我。

三、我应该怎么帮你，我需要改进的地方

我的问题1：解释太多，行动指令太少

我的回复经常包含5-7个概念的解释，然后最后一行才说"去改一下"。用户读完长解释之后注意力已经消耗，行动指令被淹没了。

改进后的模式：每次回复只讲一的、小的行动，等你做完再讲下一个。

我的问题2：没有强制执行"做完

每次你带着新问题回来（而不是修改后的代码），我都接着回答新问题，而不是先问"上次叫你做的事做了吗"。这纵容了"听懂代替动手"的模式。

改进后的模式：每次你发来新问题，我先问"上次叫你做的X做了吗，给我看"，看完才继续。

我的问题3：给了超出你当前能力的任务

那个7步填空框架在你还不知道http_client参数名的情况下给出来，注定失败。任务的粒度要比你现有能力小一步，不能大两步。

改进后的模式：每次只给一件事，这件事你用已有的知识就能完成，最多查一个新东西。
